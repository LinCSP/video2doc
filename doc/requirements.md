# Требования к приложению Video2Doc

## 1. Функциональные требования (FR)

### FR-1. Управление видеофайлом
- **FR-1.1** Пользователь должен иметь возможность выбрать видеофайл через стандартный диалог выбора файла (`wxFileDialog`).
- **FR-1.2** Поддерживаемые форматы: `.mp4`, `.mkv`, `.avi`, `.mov` и другие форматы, поддерживаемые ffmpeg.
- **FR-1.3** Выбранный путь должен отображаться в текстовом поле и сохраняться в настройках.

### FR-2. Настройка интервала скриншотов
- **FR-2.1** Интервал извлечения кадров задаётся в секундах (целое число ≥ 1).
- **FR-2.2** Значение по умолчанию: **1 секунда**.
- **FR-2.3** Поле ввода должно поддерживать валидацию (только положительные целые числа).

### FR-3. Настройка выходных путей
- **FR-3.1** Пользователь должен иметь возможность указать папку для сохранения результатов через `wxDirDialog`.
- **FR-3.2** По умолчанию используется подпапка с именем видеофайла в текущей рабочей директории.
- **FR-3.3** Путь к папке `screenshots` формируется автоматически как подпапка выходной директории.
- **FR-3.4** Все пути должны быть абсолютными и проверяться на существование/доступность.

### FR-4. Управление проектами
- **FR-4.1** В интерфейсе должен быть список (`wxListBox` или `wxListCtrl`) дополнительных проектов.
- **FR-4.2** Пользователь может добавлять проекты через диалог выбора папки.
- **FR-4.3** Пользователь может удалять выбранные проекты из списка.
- **FR-4.4** Список проектов сохраняется в конфигурации приложения.

### FR-5. Редактирование шаблона промпта
- **FR-5.1** В интерфейсе должен быть многострочный текстовый редактор (`wxTextCtrl` с флагом `wxTE_MULTILINE`) для редактирования `promt.md`.
- **FR-5.2** Шаблон должен содержать плейсхолдеры для автоматической подстановки путей:
  - `{VIDEO_PATH}` — путь к видео
  - `{OUT_DIR}` — выходная директория
  - `{SCREENSHOTS_DIR}` — папка со скриншотами
  - `{PROJECTS_LIST}` — список дополнительных проектов
- **FR-5.3** Редактор должен иметь кнопки «Сохранить шаблон» и «Сбросить к умолчанию».
- **FR-5.4** Шаблон сохраняется в файл `promt.md` в выходной директории.

### FR-6. Двухэтапный рабочий процесс
- **FR-6.1** **Этап 1 — Транскрибация**:
  - Запуск ffmpeg для извлечения кадров с заданным интервалом.
  - Запуск faster_whisper для распознавания речи и генерации `transcript.txt` / `transcript.srt`.
  - Индикация прогресса через `wxGauge`.
  - Отображение лога выполнения в реальном времени.
- **FR-6.2** **Этап 2 — Генерация документации**:
  - Формирование финального промпта на основе шаблона и подстановки путей.
  - Запуск Kimi Code CLI с сформированным промптом.
  - Передача Kimi путей к скриншотам и транскрипции.
  - Сохранение сгенерированной документации в `doc/`.
- **FR-6.3** Пользователь должен иметь возможность запустить каждый этап отдельно или оба этапа последовательно.

### FR-7. Интеграция с внешними инструментами
- **FR-7.1** Приложение должно запускать внешние процессы (ffmpeg, whisper, kimi) через `wxProcess` с перехватом stdout/stderr.
- **FR-7.2** Вывод внешних процессов должен отображаться в панели лога в реальном времени.
- **FR-7.3** Должна быть возможность прервать выполняющийся процесс.

### FR-8. Сохранение и загрузка настроек
- **FR-8.1** Все параметры (пути, интервал, список проектов, шаблон промпта) сохраняются между сессиями.
- **FR-8.2** Используется механизм `wxConfig` или JSON-файл конфигурации.

## 2. Нефункциональные требования (NFR)

| ID | Требование | Описание |
|----|------------|----------|
| NFR-1 | Платформы | Windows 10+, macOS 12+, Linux (GTK3). |
| NFR-2 | GUI-фреймворк | wxWidgets 3.2+ с нативным внешним видом. |
| NFR-3 | Производительность | Интерфейс не должен блокироваться при длительных операциях (использование `wxProcess` + события). |
| NFR-4 | Локализация | Интерфейс на русском языке, с возможностью расширения. |
| NFR-5 | Масштабируемость | Окно приложения должно корректно масштабироваться (`wxSizer`). |
| NFR-6 | Надёжность | При ошибке внешнего процесса выводится сообщение `wxMessageBox` с деталями. |
| NFR-7 | Безопасность | Пути проверяются на валидность, запрещён выход за пределы рабочей директории. |

## 3. Пользовательские сценарии

### Сценарий 1: Базовый двухэтапный запуск
1. Пользователь запускает приложение.
2. Выбирает видеофайл через «Обзор».
3. Указывает интервал скриншотов (например, 1 сек).
4. Проверяет/корректирует выходную папку.
5. При необходимости добавляет дополнительные проекты в список.
6. Редактирует шаблон промпта в соответствии с задачей.
7. Нажимает «Запустить полный цикл».
8. Приложение выполняет Этап 1 (скриншоты + транскрипция).
9. По завершении Этапа 1 автоматически запускается Этап 2 (генерация документации через Kimi).
10. Результат сохраняется в `doc/` выходной директории.

### Сценарий 2: Повторная генерация с новым промптом
1. Пользователь открывает существующий проект (уже есть скриншоты и transcript).
2. Изменяет шаблон промпта.
3. Нажимает «Только Этап 2 — Генерация документации».
4. Kimi перегенерирует документацию на основе нового промпта.

## 4. Ограничения и допущения

- Для работы требуется установленный **ffmpeg** и доступ к **faster_whisper** (Python-окружение).
- **Kimi Code CLI** должен быть установлен и доступен в переменной окружения `PATH`.
- Приложение не выполняет собственную транскрибацию «с нуля», а управляет существующими инструментами.
- Обработка видео большого размера может занимать значительное время — интерфейс должен оставаться отзывчивым.
